Architecture Summary

This project has one runtime entry function and one entity class. The main loop drives events, updates each square, and renders every frame. The Square methods split movement into focused steps: flee force, jitter, speed limit, movement, and wall bounce.

Good Patterns
[+] Clear Behavior Decomposition

The update path is split into apply_behaviors, apply_jitter, limit_speed, move, and bounce, which improves readability and testing.

Basics: Small methods are easier to reason about.
Engineering Insight: Each method has a single purpose, so regressions are easier to isolate.
Architecture Insight: This resembles a mini update pipeline for entity systems.
Read about method size and decomposition
[+] Centralized Simulation Constants

Global constants define window size, timing, and force tuning in one place, so behavior changes are predictable.

Basics: Constants avoid repeating magic values.
Engineering Insight: Parameter tuning can happen without touching core logic.
Architecture Insight: Configuration is separated from algorithm steps.
Python module constants
[+] Frame Time Normalization

Using dt from clock.tick makes movement scale with time rather than frame count.

Basics: Motion stays similar at different FPS values.
Engineering Insight: Time-based updates reduce hardware-dependent behavior drift.
Architecture Insight: This supports deterministic simulation steps and future interpolation strategies.
Pygame clock timing docs
Potential Issues
[!] Tight Coupling in Entity Methods

Square physics, wall collisions, and logging are mixed in the same class. This can make feature growth harder.

Basics: Too many responsibilities make classes harder to maintain.
Engineering Insight: Logging side effects in movement code complicate testing and performance profiling.
Architecture Insight: Consider separating physics update from instrumentation concerns.
Improvement hint: pass bounce events to a higher-level reporter instead of writing logs inside entity motion.
[!] Unlabeled Force Scaling Value

The factor * 100 in flee acceleration is a hidden tuning value without a named constant.

Basics: Hidden numbers make code intent unclear.
Engineering Insight: Contributors can accidentally break balance when they cannot trace force units.
Architecture Insight: Domain constants should be explicit to stabilize future gameplay tuning.
Improvement hint: replace 100 with a named constant like FORCE_SCALE.
